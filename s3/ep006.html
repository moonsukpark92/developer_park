01 · 새로운 세계

GitHub는
블로그가 아니었다

AI와 일하다 보니 자연스럽게 GitHub에 들어가게 됐다. 처음에는 개발자들의 블로그 정도로 생각했다. 코드 올려놓는 곳이겠지 싶었다.

막상 들어가 보니 전혀 달랐다. 공식 문서보다 더 실전적인 코드가 있고, 누군가 이미 만들어 놓은 소스코드가 있고, 비슷한 문제를 먼저 해결한 사람들의 흔적이 있었다. 어마어마한 곳이었다.

모르면 GitHub를 뒤지고,
Anthropic 공식 문서를 뒤지고,
관련 소스코드가 있으면 다운받아
분석해서 현재 프로젝트에 반영한다.
그랬더니 코드의 수준이 달라졌다.

02 · 방법의 차이

그냥 시키는 것과
전략적으로 시키는 것

같은 Claude Code를 쓰더라도 어떻게 쓰느냐에 따라 결과물이 완전히 달라진다. 직접 비교해보고 나서 확실히 깨달았다.

방법 A — 단순 위임
그냥 만들어줘
Claude Code에게 요구사항만 전달
결과물 실행해보고 확인
에러 나면 고쳐달라고 반복
왜 이렇게 만들어졌는지 모름
결과물 퀄리티: 기본 수준
방법 B — 전략적 접근
리소스를 먼저 수집하고 시킨다
기본 계획 수립
GitHub · Anthropic 문서 · 인터넷 서칭
유사 소스코드 다운로드 및 분석
현재 프로젝트에 핵심 내용 반영
Claude가 키포인트를 찾아 업데이트
결과물 퀄리티: 현저히 향상

방법 B로 진행하면 Claude Code가 기가 막히게 핵심 포인트를 찾아낸다. 처음 계획했던 것보다 훨씬 빠른 속도로 코드가 업그레이드되고, 프로그램의 완성도가 올라간다.

03 · 수집해야 할 세 가지

시작 전에
이 세 곳을 먼저 뒤진다

계획을 세웠으면 바로 코딩을 시키지 않는다. 먼저 이 세 가지 리소스를 수집한다. Claude에게 가져오라고 하면 알아서 찾아온다.

🐙
01
GitHub
유사한 기능의 오픈소스 소스코드. 이미 누군가 만들어 놓은 것을 참고하면 시간이 절반으로 줄어든다.
📖
02
Anthropic 공식 문서
공식 문서에 정답이 있다. Claude가 직접 읽고 요약해서 가져온다. 가장 정확한 기술 출처.
🌐
03
인터넷 서칭
유사 성능의 제품, 최신 구현 방법, 커뮤니티에서 검증된 방식. 세상이 이미 알고 있는 것을 활용한다.

이 세 가지에서 핵심 기술과 핵심 코드를 모두 가져온 다음, 현재 진행 중인 프로젝트에 반영한다. 그 과정을 거치고 나서 Claude Code에게 업데이트를 지시한다. 결과물의 수준이 전혀 달라진다.

04 · 자기 자신을 검토하게 한다

코드를
믿지 마라

Claude Code가 만들어준 코드를 그냥 믿으면 안 된다. 돌아가면 됐다고 생각하면 안 된다. 스스로 검토하게 해야 한다.

방법이 있다. Claude에게 베타 테스터 또는 코드 전문가 페르소나를 부여하고, 지금까지 만든 코드와 프로그램을 심도 있게 검토해서 보고서 또는 MD 파일 형식으로 만들어달라고 하는 것이다.

페르소나 부여

"너는 시니어 개발자야. 이 코드를 베타 테스터 관점에서 심도 있게 검토해줘."

보고서 작성

계획서와 현재 코드를 대조해 보고서 또는 MD 파일로 만들어달라고 한다. 꼼꼼하게 읽어야 한다.

점수 평가

100점 만점으로 현재 완성도를 점수화한다. 31점이면 개념적 수준, 61점이면 실제 사용 가능한 소프트웨어 수준이라는 식으로 판단한다.

개선 계획 수립

100점에 가깝게 만들려면 무엇이 필요한지 집중한다. 그 내용으로 계획서를 만든다.

계획서 검토 · 질문 · 반영

계획서를 꼼꼼히 읽고 이해가 안 되는 부분은 다시 물어본다. 이해가 된 다음 최종 업데이트를 지시한다.

보고서를 받아 읽다 보면 내가 몰랐던 인사이트, 몰랐던 정보들이 상당히 많이 담겨 있다. 그걸 이해하고 나서 업데이트하면 프로그램의 퀄리티가 현저히 좋아진다는 것을 깨달았다.

05 · 기록의 힘

개발 일지를
MD로 남긴다

그냥 vibe coding을 한다고 해서 딸깍 실행하고 결과만 보는 것으로 끝내면 안 된다. 어떻게 만들었고 어떤 과정을 거쳤는지, 개발 일지를 MD 파일로 저장하게 해야 한다.

기록이 쌓이면 다음 번에 같은 문제를 마주쳤을 때 훨씬 빠르게 대응할 수 있다. 내가 어디까지 왔는지도 보인다. 그 과정 자체가 실력이 된다.

딸깍 한 번으로 끝나는 개발은 없다.
계획하고, 리소스를 수집하고,
검토하고, 기록하고, 다시 업데이트한다.
그 반복이 프로그램의 수준을 만든다.
