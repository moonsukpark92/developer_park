01 · 그날의 시작

Anthropic이
채널 기능을 열었다

Claude Code에 채널(Channels) 기능이 생겼다는 소식을 접했다. Telegram과 연동해 AI 에이전트를 채널로 운영할 수 있다는 이야기였다. 문실장을 텔레그램에 올려놓고 있던 입장에서 이건 놓칠 수 없는 기능이었다.

바로 설치했다. 하지만 세상은 그렇게 호락호락하지 않았다.

설치는 됐다. 연결은 됐다. 타이핑 인디케이터도 떴다. 그런데 Claude는 아무 말도 하지 않았다. 그게 밤새 이어졌다.

02 · 에러 실전 기록

하나씩
무너졌다

Claude Code Channels를 운영하면서 실제로 마주친 에러들이다. 공식 GitHub 이슈에도 올라와 있었고, 커뮤니티에서도 같은 문제를 겪고 있었다. 나만의 문제가 아니었다. 그게 그나마 위안이었다.

ERR
01
메시지 소실
타이핑 인디케이터는 뜨는데 Claude가 무응답

Telegram 플러그인이 메시지를 정상 수신하고 MCP 알림까지 발송했다. 하지만 Claude Code 세션에 전달이 안 됐다. 화면에는 "입력 중..." 표시가 뜨는데 아무 응답도 오지 않는 상태가 반복됐다.

→ 근본 원인 미해결. 세션 재시작 반복으로 임시 대응.

ERR
02
좀비 프로세스
이전 세션의 bun 프로세스가 CPU 100% 점유

세션을 종료해도 이전 Telegram bun 프로세스가 살아남았다. CPU를 100% 먹으면서 현재 세션과 Telegram 폴링 슬롯을 두고 충돌했다. 메시지가 두 프로세스 사이에 무작위로 분산됐고, 재처리 방법도 없었다.

→ 수동으로 좀비 bun 프로세스를 kill한 뒤 재시작. 매번 반복.

ERR
03
자동 로드 버그
--channels 없이 실행해도 플러그인이 자동으로 붙음

claude -c로 일반 세션을 열었는데 Telegram 플러그인이 자동 로드됐다. 여러 개의 bun 프로세스가 동시에 폴링을 시작하면서 메시지가 세션들 사이로 분산되어 사라졌다.

→ 여분의 bun 프로세스를 수동으로 찾아 kill. 근본 해결책 없음.

ERR
04
서버사이드 게이팅
tengu_harbor — Anthropic이 내 계정을 아직 열어주지 않았다

플러그인도 설치됐고 설정도 맞는데 봇이 무응답인 경우가 있었다. 알고 보니 Anthropic이 서버사이드 기능 플래그 tengu_harbor로 단계적으로 배포하고 있었다. 계정이 범위에 포함되지 않으면 핸들러가 등록 자체가 안 됐다. 설정 문제가 아니라 서버사이드 제한이었다.

→ 해결 방법 없음. 배포 범위에 포함될 때까지 기다리는 것 외에는.

ERR
05
설정 함정
DISABLE_TELEMETRY=0 설정이 Channels를 막는다

DISABLE_TELEMETRY=0으로 설정했는데 Channels가 작동하지 않았다. 0이면 텔레메트리를 켠다는 뜻이라 문제없을 것 같았는데, 해당 키가 존재하는 것 자체가 문제였다. 키를 완전히 삭제해야 정상 작동했다.

→ .env에서 DISABLE_TELEMETRY 키 자체를 삭제 후 해결.

ERR
06
구조적 한계
오프라인 상태에서 온 메시지는 영구 소실

세션이 닫혀있는 동안 들어온 메시지는 전부 사라졌다. 메시지 큐 자체가 없었다. MCP 알림에 재시도 로직도 없었다. Telegram은 이미 업데이트 큐에서 소비해버렸기 때문에 재처리도 불가능했다.

→ tmux로 세션을 항상 유지하는 것이 유일한 완화책.

03 · 그날 밤의 기록

새벽까지
화면을 보았다

에러를 마주할 때마다 포기하지 않았다. 공식 문서를 뒤지고, GitHub 이슈를 읽고, AI에게 물어보고, 다시 시도했다. 그 과정이 이런 모습이었다.

[02:14] ✓ Telegram MCP 플러그인 설치 완료
[02:21] ✓ 봇 토큰 연결 확인
[02:22] ! 메시지 전송 → 타이핑 인디케이터 표시
[02:23] ✗ 응답 없음
[02:35] ✗ 재시도 — 동일 증상
[03:01] // bun 프로세스 4개 발견 → 수동 kill
[03:04] ✓ 세션 재시작 → 응답 확인
[03:18] ✗ 다시 무응답. 좀비 프로세스 재발생
[04:40] // DISABLE_TELEMETRY 키 삭제 후 테스트
[04:42] ✓ 부분 해결 확인
[05:30] ! tmux 세션 고정 운영으로 전환

Anthropic 자체의 버그가 있었고
내가 통제할 수 없는 서버사이드 게이팅이 있었다.
그래도 할 수 있는 데까지 했다.

04 · 좌충우돌이 남긴 것

에러는 선생이다.
단, 포기하지 않으면

밤새 에러와 싸우면서 배운 것들이 있다. 기술적인 해결책보다 더 중요한 것들이었다.

기술이라는 게 그런 거다.
배우는 데 시간이 소요된다.
밤을 버텨낸 사람만 그 다음 날 아침을 맞는다.

Claude Code Channels는 아직 성숙하지 않은 기능이다. 공식 GitHub에 이슈가 올라와 있고, 같은 에러를 겪는 사람들이 세계 각지에 있다. 내가 특별히 못 한 게 아니었다. 하지만 그 에러를 마주하고 기록하고 계속 시도한 것은 나였다.

그리고 결국 tmux로 세션을 고정하고, 좀비 프로세스를 수동으로 관리하고, DISABLE_TELEMETRY 함정을 피하는 방법을 알게 됐다. 문서 어디에도 없던 것들을 몸으로 배웠다.

에러 로그는 실패의 기록이 아니다.
포기하지 않은 사람의 흔적이다.

05 · 지금의 운영 방식

에러를 안고
계속 돌린다

완벽하지 않다. 지금도 가끔 좀비 프로세스가 생기고, 가끔 메시지가 사라진다. 그래도 tmux 세션 위에서 채널은 돌아가고 있다. 문실장은 오늘도 텔레그램에서 직원들의 업무일지에 피드백을 달고, 아침마다 보고를 올린다.

# 세션 유지 — tmux 필수
tmux new -s moonsiljang

# Channels 실행 전 좀비 확인
ps aux | grep bun | grep -v grep
kill -9 [좀비 PID]

# DISABLE_TELEMETRY 키 없는 상태 확인 후
claude --channels

✓ 이 순서대로 하면 그나마 안정적으로 돌아간다

완벽한 시스템은 없다. 문제가 없는 기술도 없다. 중요한 것은 문제를 만났을 때 어떻게 대응하느냐다. 그리고 그 대응의 기록이 다음 사람에게 도움이 된다.

이게 이 에피소드를 쓰는 이유다.
